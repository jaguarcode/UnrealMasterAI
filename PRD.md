Unreal Engine 자율형 AI 에이전트 개발 프로젝트 PRD

# 제품 요구사항 문서 (PRD): 차세대 Unreal Engine 자율형 AI 에이전트 개발 프로젝트

\*\* 아래의 내용은 초안으로서 작성된 PRD 문서로, 충분한 조사와 연구를 통해서 프로젝트의 아키텍트를 고도화해야 함. 핵심 기술 스택 및 개발 단계는 언제든지 변경 가능.

## 1. 프로젝트 개요

### 1.1. 배경 및 목적

현재 글로벌 게임 개발 산업은 AI 기술 도입으로 패러다임이 전환되고 있으며, 대형 언어 모델(LLM)을 활용한 개발 워크플로우의 자동화가 필수로 자리 잡고 있습니다 [1]. 그러나 범용 LLM 코딩 어시스턴트는 Unreal Engine(이하 UE) 특유의 복잡한 C++ 매크로, UObject 시스템, Blueprint 시각적 스크립팅, Slate 및 UMG 프레임워크 등을 완벽히 이해하고 제어하는 데 심각한 한계를 보이고 있습니다 [2]. 기존 도구들(Ludus AI, Aura AI 등) 역시 Blueprint의 심층적 텍스트 변환, 순수 C++ 기반 Slate UI 생성, 완전한 자율 컴파일 사이클 측면에서 한계가 명확합니다 [3, 4].

본 프로젝트의 목적은 **Claude Code(Claude 3.5 Sonnet)와 Model Context Protocol(MCP)을 결합하여 엔진의 내부 API와 양방향으로 통신하고, Blueprint 로직 생성 및 C++ 기반 Slate UI 코딩부터 Live Coding 컴파일까지 자율적으로 수행하는 완전한 '가상 개발 파트너' 에이전트를 개발**하는 것입니다 [2, 5].

### 1.2. 핵심 가치 제안

- **Blueprint의 양방향 직렬화 및 물리적 제어:** 메모리 단에서 Blueprint 노드를 생성하고 핀을 연결 [6].
- **고품질 Slate UI 자율 생성:** 프롬프트 엔지니어링 및 RAG를 활용한 문법적 한계 돌파 [7].
- **자율 컴파일 및 자가 치유 루프:** C++ 코드 작성 후 수동 개입 없이 즉각적으로 Live Coding을 트리거하고 오류 발생 시 스스로 분석하여 수정 [8].
- **관측성(Observability) 플랫폼 및 안전장치 도입:** 에이전트가 자가 치유 루프를 수행하는 동안 LangSmith나 Langfuse와 같은 관측성 플랫폼을 연동하여 실행 추적(Trace) 데이터를 수집하면, 에이전트의 에러 패턴과 도구 사용 통계를 더욱 정밀하게 분석하고 최적화할 수 있습니다 . 또한, 에이전트가 엔진 내의 중요한 에셋을 수정하거나 삭제할 때는 예기치 않은 오류를 방지하기 위해 반드시 개발자의 승인을 거치도록 하는 'Human-in-the-loop(인간 개입)' 권한 제어 단계를 아키텍처에 포함하는 것이 안전합니다 .
- **동적 도구 검색(Semantic Retrieval) 계층 구축:** 에이전트의 기능이 확장되어 활용해야 할 도구와 API 문석가 수십~수백 개로 늘어날 경우를 대비하여, 모든 도구의 스키마를 LLM에 전달하기보다는 사용자의 현재 명령 맥락에 가장 적합한 도구만 실시간으로 검색하여 제공하는 의미론적 검색(RAG) 계층을 도입하면 컨텍스트 윈도우 소모를 획기적으로 줄일 수 있습니다

## 2. 시스템 아키텍처 및 기술 스택

본 프로젝트는 상호 유기적으로 연결된 4가지 주요 계층으로 설계됩니다 [9].

### 2.1. 계층별 아키텍처 구조

1. **MCP Host (Claude Code):** 사용자의 쿼리를 해석하고 도구 호출 순서를 결정하는 추론의 중추 역할을 담당합니다 [9, 10].
2. **MCP Client:** Claude Code 내부에 위치하며 외부 서버와의 JSON-RPC 통신을 중재합니다 [9].
3. **MCP Server (Node.js/TypeScript 기반 브릿지 서버):** `stdio` 방식으로 Claude Code와 통신하며, UE와 클라이언트 간의 실질적인 브릿지 역할을 수행합니다 [9-11].
4. **UE Agent Plugin (C++ 및 Python):** 엔진 내부에 위치하며 WebSockets를 통해 브릿지 서버와 통신하고, 엔진의 내부 상태 제어 및 리소스를 조작합니다 [10].

### 2.2. 핵심 기술 스택

| 구성 요소              | 기술 스택                       | 선정 이유                                                                                                  |
| :--------------------- | :------------------------------ | :--------------------------------------------------------------------------------------------------------- |
| **LLM Inference**      | Claude 3.5 Sonnet / Claude Code | 복잡한 C++ 코드 생성 및 방대한 엔진 API 컨텍스트 추론에 압도적 성능 입증 [10].                             |
| **MCP Bridge Server**  | Node.js (TypeScript)            | 로컬 `stdio` 통신 최적화, 정적 타입 시스템 통한 메시지 안전성 및 비동기 소켓 처리 용이 [10].               |
| **통신 프로토콜**      | WebSockets (via TCP/IP)         | 엔진과 브릿지 간의 영구적 상태 유지 및 실시간 양방향 푸시(Push) 환경 제공 [10, 12].                        |
| **UE Plugin Core**     | C++ (UE 5.x Native)             | UEdGraphPin 접근, Slate UI 델리게이트 바인딩, ILiveCodingModule 호출 등 저수준 메모리 제어에 필수 [6, 10]. |
| **Scripting Helper**   | Python API                      | 에셋 파싱, 디렉토리 구조 생성 등 메타 작업을 유연하게 처리 [10, 13].                                       |
| **Data Serialization** | JSON                            | 복잡한 Blueprint 구조를 단순화하여 LLM 토큰 부담 완화 및 환각 억제 [10, 14].                               |

## 3. 주요 기능 요구사항

### 3.1. 컨텍스트 및 통신 최적화

- **상태 보존형(Stateful) 도구 처리:** 방대한 Blueprint 데이터나 액터 리스트를 그대로 반환하여 토큰을 낭비하지 않고, 키-값 저장소에 원시 데이터를 캐싱한 뒤 식별자만 LLM에 반환해야 합니다 [15].
- **안전한 통신 및 로깅:** MCP Server 통신 시 JSON-RPC 스트림 손상을 막기 위해 모든 디버그 로그는 반드시 표준 에러(`stderr`)로 우회 처리해야 합니다 [11].

### 3.2. 자율형 Blueprint 변환 및 제어 메커니즘

- **AST 기반 JSON 직렬화:** 장황한 T3D 포맷 대신, Blueprint의 추상 구문 트리를 논리적 실행 흐름 중심의 JSON 포맷으로 구조화하여 추출하고 주입할 수 있어야 합니다 [14, 16].
- **C++ 기반 동적 노드 조작:** Python API의 한계(UFUNCTION 래핑 함수만 접근 가능)를 극복하기 위해 C++ 레벨에서 `UEdGraphPin` 클래스 인터페이스(`MakeLinkTo`, `BreakLinkTo`)를 직접 활용하여 핀을 연결하고 노드를 스폰하는 래퍼 함수를 구현해야 합니다 [6, 13].

### 3.3. Slate UI 프레임워크 C++ 개발 고도화

- **다이나믹 템플릿 인젝션(RAG):** C++의 선언적 구문에 익숙하지 않은 LLM을 위해, Slate 예제 코드와 베이스 템플릿 마크다운 텍스트를 LLM의 컨텍스트에 삽입해야 합니다 [7, 17].
- **스타일 가이드라인 강제:** 시스템 프롬프트를 통해 `TAttribute` 델리게이트 폴링 활용, 매크로 내 `SNew`/`SAssignNew` 규칙 등 명시적인 제약 조건을 주입해야 합니다 [17, 18].

### 3.4. Live Coding 자동화 및 자가 치유 (Self-Healing)

- **Live Coding 자율 트리거:** 코드가 작성/수정된 후 플러그인이 `ILiveCodingModule-&gt;Compile()`을 백그라운드 스레드에서 자동 호출해야 합니다 [19].
- **컴파일 상태 파싱 및 자가 복구:** 컴파일 콜백 리스너를 구현하여 출력 로그를 후킹, 에러 발생 시 로그를 분석하여 코드를 재수정하는 자가 치유 루프를 실행해야 합니다 [8].

## 4. 단계별 시스템 구현 로드맵 (4단계)

### Phase 1: 통신 인프라 및 베이스 컨텍스트 구축 [20]

1. Node.js/TypeScript 환경에서 `@modelcontextprotocol/sdk`를 사용하여 `stdio` 기반 MCP 서버 구축.
2. UE 5.x 내부에 백그라운드 스레드로 동작하는 비동기 WebSocket 서버 플러그인 초기화 및 개발.
3. Claude Code `.mcp.json`에 서버 등록 및 기초 쿼리(현재 레벨 조회 등)를 통한 엔드투엔드(E2E) 통신 검증.

### Phase 2: Blueprint 직렬화 및 물리적 조작 엔진 구현 [21]

1. C++ 플러그인 내에 Blueprint 논리 실행 그래프 순회 및 속성/핀 정보를 JSON으로 직렬화하는 파싱 모듈 개발.
2. 외부 JSON 페이로드 명령에 따라 `FKismetEditorUtilities`와 `UEdGraphPin`을 활용하여 노드를 스폰하고 핀을 연결하는 C++ 로직 구현.
3. LLM의 명령(예: "특정 이벤트 노드 연결")이 에디터 창에 즉각적으로 반영되는지 검증.

### Phase 3: Slate UI 컨텍스트 생성기 및 Live Coding 융합 [22]

1. 로컬 파일 시스템 내 엔진 표준 코딩 규약에 맞춘 Slate UI 마크다운 문서를 비치하고 File Search 도구와 연동하여 RAG 시스템 구축.
2. `ILiveCodingModule` 제어용 C++ 래퍼 함수를 MCP 도구 목록에 등록.
3. 컴파일 출력 로그 스트림 후킹 및 콜백 수집 모듈 구현. LLM을 통한 Custom Slate Widget 작성 및 Live Coding 팝업 검증.

### Phase 4: 에이전트 자율성 심화 및 시스템 워크플로우 최적화 [23]

1. 컴파일 에러 경고 로그를 정제하여 LLM에게 재전송하는 에러 복구(Self-healing) 로직 고도화.
2. 컨텍스트 윈도우 한계 극복을 위한 마이크로 툴(필요한 부분만 탐색) 체계 적용.
3. UE 에디터 내부에 WebBrowser 컴포넌트를 활용한 독립형 채팅 패널 플러그인 부착하여 사용자 경험(UX) 개선 및 통합.
